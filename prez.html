le problème

La confiance, une fois acquise,
ne s'arrête jamais

Un accès, un mot de passe, une confiance réseau : une fois obtenus, ils restent valides indéfiniment.

01
Une clé SSH ne change jamais, est parfois partagée entre collègues, et reste valide tant que personne n'y pense.
02
Un mot de passe de base de données dans un fichier de config reste valide même après une fuite de logs ou un commit accidentel sur un dépôt Git.
03
Un serveur fait confiance à un autre simplement parce qu'il est dans le même réseau : une intrusion donne accès à tout.
le principe

Zero-trust

Aucune confiance n'est jamais acquise.
Chaque accès est vérifié à l'instant T,
et expire par défaut.
vue d'ensemble

Trois briques, trois mécanismes de confiance

secret applicatif
Vault
Une appli a besoin d'un mot de passe, qui n'existe pas avant la demande.
identité machine
SPIRE
Une machine prouve qui elle est par certificat plutôt que par mot de passe.
accès humain
Boundary
Un ingénieur veut atteindre un serveur sans jamais connaître son adresse.

vault, 2 / 2. cheminement
brique 1. vault

Un compte de base qui naît et meurt

Aucun mot de passe stocké, Vault crée et révoque le compte.

vault, 2 / 2. démo
démo live

make demo-db

Génère des credentials PostgreSQL éphémères, vérifie leur existence dans la base, puis révoque le lease en direct.
vault read database/creds/app-role
→ username: v-root-app-XxXx   lease_duration: 1h

vault lease revoke <lease_id>
# → l'utilisateur n'existe plus dans PostgreSQL
